     Consent-Fluss

     1. AppSettings hГӨlt AnalyticsEnabled (bool, default true),
     CrashReportingEnabled (bool, default true), PrivacyBannerDismissed (bool,
     default false), PrivacyPolicyAcceptedAt (DateTime?).
     2. AnalyticsConsentGate (Core, Singleton) liest Consent aus
     IAppSettingsProvider, wartet via
     DatabaseInitializationHelper.EnsureInitializedAsync() auf DB-Init, und
     abonniert SettingsChanged aus AppSettingsProvider.cs:21.
     3. Bei jedem Flip feuert ConsentChanged; Android-Services rufen synchron
     FirebaseAnalytics.Instance.SetAnalyticsCollectionEnabled(bool) bzw.
     FirebaseCrashlytics.Instance.SetCrashlyticsCollectionEnabled(bool).
     4. ZusГӨtzliche Defense-in-Depth: jeder LogEvent-Call auf dem
     Android-Service short-circuitet, wenn _gate.AnalyticsAllowed == false.
     5. Bei Analytics-Opt-Out zusГӨtzlich
     FirebaseAnalytics.Instance.ResetAnalyticsData() вҶ’ Instance-ID rotiert.

в—Ҹ Crashlytics "Build ID" вҖ” was das ist und warum es in MAUI wackelt

  Worum geht's beim "Build ID"

  Firebase Crashlytics vergibt pro Release-Build eine eindeutige ID, die (a) in
  die APK eingebettet wird und (b) an die vom R8-Compiler erzeugte Mapping-Datei
   gekoppelt wird, die Crashlytics auf seinen Server hochlГӨdt. Wenn spГӨter ein
  Crash-Report reinkommt, sucht Firebase anhand der Build-ID die passende
  Mapping-Datei und Гјbersetzt kryptische Stack-Frames zurГјck in lesbare Klassen-
   und Methodennamen.

  Das funktioniert im Android-Г–kosystem Гјber ein Gradle-Plugin
  (com.google.firebase.crashlytics), das beim assembleRelease automatisch (a)
  die ID erzeugt und (b) die Mapping-Datei hochlГӨdt.

  Warum das in .NET MAUI bricht

  .NET MAUI baut Android-Apps nicht mit Gradle, sondern mit MSBuild. Das
  Gradle-Plugin lГӨuft also nie вҖ” es gibt keinen MSBuild-Г„quivalent-Task, der
  Build-ID + Mapping-Upload Гјbernimmt. Folgen:

  - Die APK enthГӨlt keinen com.crashlytics.build.id-Marker.
  - Die R8-Mapping-Datei wird nicht automatisch zu Firebase hochgeladen.
  - Crashlytics loggt beim Start eine Warnung wie "Crashlytics build ID is 
  missing" und schaltet in einen Best-Effort-Modus.

  Das ist ein bekanntes Upstream-Issue bei Microsoft/.NET MAUI вҖ” siehe die
  Q&A-Links in SECURITY.md. Es ist kein Fehler in unserer Integration.

  Ich habe den kritischen Code-Pfad gefunden. Die Crash-Ursache ist
  hГ¶chstwahrscheinlich eine DI-Reihenfolge-Falle:

  Startup-Sequenz:
  1. MainApplication.OnCreate() (via MauiApplication) вҶ’ triggert CreateMauiApp()
  2. MauiProgram.CreateMauiApp() baut DI вҶ’ Zeile 48:

  3. Dessen Konstruktor ruft FirebaseCrashlytics.Instance вҶ’ ruft intern
  FirebaseApp.getInstance() вҶ’ wirft IllegalStateException wenn FirebaseApp noch 
  nicht initialisiert
  4. MainActivity.OnCreate() вҖ” wГјrde Firebase.FirebaseApp.InitializeApp(this)

  Der CrashlyticsLoggerProvider in ConfigureLogging hat dieselbe Falle: sobald
  irgendein ILogger<T> resolved wird, triggert die Factory
  GetRequiredService<ICrashReportingService>().

  Warum Firebase auto-init nicht greift: Firebase registriert normalerweise
  einen FirebaseInitProvider (ContentProvider mit hoher PrioritГӨt) via
  Manifest-Merge, der VOR Application.OnCreate lГӨuft. In
  Xamarin.Firebase-Bindings auf .NET 10 ist der Manifest-Merge-Durchlauf
  fragiler вҖ” das ist ein bekanntes Problem.

  Fix-Strategie: Firebase-Services lazy machen вҖ” .Instance / .GetInstance()
  NICHT im Konstruktor aufrufen, sondern beim ersten Zugriff. Falls auto-init

  Das macht die Services gegen beide FГӨlle robust: auto-init funktioniert вҶ’
  alles gut; auto-init funktioniert nicht вҶ’ lazy + Fallback greift.
